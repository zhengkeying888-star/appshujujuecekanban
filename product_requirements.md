## 1. 协作协议（人与 AI）

### 1.1 角色定义

| 角色 | 身份 | 职责 | 禁区 |
|------|------|------|------|
| **用户** | 业务方（运营统筹） | 提供数据源、确认业务规则、验收数据准确性、决策策略方向 | 不修改 Python 脚本，不手动编辑生成的 HTML |
| **Claude** | 工程方 | 编写/维护分析脚本与看板生成器，执行数据清洗与验证，修复 bug | **不擅自修改已确认的业务规则** |

### 1.2 协作契约（不可违背）

#### 契约一：规则变更必须经用户明确授权

**规则**：`TARGET_RESOURCES`、`AD_NAME_MAP`、`价格带映射`、`品类映射` 等已确认的业务规则，未经授权严禁修改。

**Why**：用户明确声明「过往的规则，除非我说改，你才能改，否则别动我的规则」。擅自修改导致瀑布图和效率表完全失真。

**How to apply**：
- 任何涉及映射表的修改，必须先问用户「是否需要调整」
- 即使代码上「看起来更合理」，只要用户没有明确说改，就保持原样
- 这些映射表是业务契约（business contract），不是技术优化点

#### 契约二：命中关键改动必须验证

**规则**：修改 `generate_analysis.py` 或 `generate_dashboard_v2.py` 后，必须验证下游 JSON 和 HTML 的关键字段。

**Why**：修改后曾出现 `category_summary` 为空导致 Screen 3 看不到；也曾出现硬编码旧数据导致看板显示错误值。

**How to apply**：
- 运行 `generate_analysis.py` 后检查：
  - `monthly_summary`（线索数、GMV、转化率）
  - `core_diagnosis.signals`
  - `category_summary`、`category_detail`、`category_price_band`
  - `resource_efficiency`
- 重新生成 HTML 后，grep 检查是否包含已知的旧数值（如 `16037`、`193.5`）
- 对照「数据验证基准」表快速校验核心指标

#### 契约三：非简单任务先 plan 再执行

**规则**：复杂任务（>2 文件或涉及架构）先给出方案，用户确认后再写代码。

**Why**：避免返工。用户倾向于精准修复，不喜欢附带大量无关重构。

**How to apply**：
- 涉及新增模块、修改指标口径、调整映射表 → 先给方案
- 用户简短指令（如「继续」「git 上述版本」）→ 直接执行，不做冗余确认
- 不要每完成一步就汇报进度，完成后统一给结果

#### 契约四：文档与代码同步

**规则**：任何功能变更需同步更新 `product_requirements.md` 及 `CLAUDE.md`。

**Why**：PRD 和 CLAUDE.md 是开发与验收的唯一依据，文档滞后会导致后续迭代口径混乱。

**How to apply**：
- 修改代码的同时修改 PRD 对应章节
- 发现新约束时立即追加到 CLAUDE.md「踩坑记录」
- 业务规则变更必须同步更新两份文档

### 1.3 沟通风格

- **全程中文**，技术术语后补中文注释（如 "Pinia（状态管理）"）
- **直接、高效、少寒暄**。用户指令通常简短，回复应直奔主题
- **不反问「Should I continue?」**：用户要求直接执行计划，不要中途停下来请示
- **验证驱动**：上传文件后检查 GMV 是否正确，部署后截图反馈实际效果

### 1.4 记忆管理

- **全局记忆**：跨项目长期偏好写入 `~/.claude/memory-log/`
- **项目记忆**：当前项目上下文写入 `~/.claude/projects/.../memory/`
- **文档记忆**：项目规则与踩坑写入 `CLAUDE.md`
- **何时记录**：用户明确说「记下来」「记住」时立即保存；涉及协作方式演化、重要系统变化时主动记录

## 2. 系统概述

### 2.1 背景

基于 `APP线索广告位拆解 3-4月更新版.xlsx`（约 3.6 万行、37 列明细数据）构建的单文件 HTML BI 看板，支持 3月/4月/3-4月对比三种视图切换与一键导出 HTML 月报。2026-05 迭代新增「周视图」模块，支持月度中期 MTD 进度汇报。

### 2.2 核心诉求转变

从「完课率-centric」转向「APP产值归因-centric」。只有跟 APP 产值强相关的指标才是核心指标。复盘目的是向上汇报，需要输出结论、支持交互下钻、自动生成策略建议。

### 2.3 业务目标

1. 让管理层 30 秒看懂「上个月 APP 产值怎么了」（一页纸诊断）
2. 区分「前端流量入口效率」和「后端变现效率」，避免把 CTR 下滑和产值下滑混为一谈
3. 支持下钻到「资源位 × 价格带 × 品类类型」三维交叉，定位产值下降的具体病灶
4. 基于下钻结果，自动生成可落地的策略建议与行动清单
5. **新增**：月度中期自动生成 MTD 进度周报，追踪目标差距与同比归因

### 2.4 交付物清单

| 交付物 | 形式 | 使用场景 |
|--------|------|---------|
| `dashboard/v2/index.html` | 单文件 HTML（含 CSS/JS/数据） | 本地浏览器打开，支持交互与月份切换 |
| `output/chart-html/weekly-report-xxx/` | 图表项目文件夹（HTML + PNG） | 周报可视化，可插入飞书文档 |
| `weekly_report_data.json` | 结构化 JSON | 周报数据源，可被看板内嵌读取 |
| `data_analysis_output.json` | 结构化 JSON | 月报数据源，被看板内嵌读取 |

## 3. 数据源与字段映射

### 3.1 主数据源

**文件**：`APP线索广告位拆解 3-4月更新版.xlsx`（又名 `APP线索广告位拆解 3-4月明细版本.xlsx`）
**规模**：35,558 行（过滤合计后），37 列

| 字段 | 说明 | 用途 |
|-----|------|------|
| `stat_month` | 统计月份（2026-03 / 2026-04 / 合计） | 月份切分与过滤 |
| `tag_level_1` | 投放位置名称（原 `广告资源位`） | 资源位效率分析 |
| `category_name` | 品类名称 | 品类分析、品类类型映射 |
| `camp_name` | 活动/课程名称 | 卖点分析回退数据源（原 `广告位素材` 已移除） |
| `sku_price` | 课程价格 | 价格带映射（0元/1.1元/3.9元/其他） |
| `线索数` | 投放获客数 | 核心指标 |
| `首单数` | 成交订单数 | 转化率分子 |
| `首单流水` | 成交金额（GMV） | 核心指标 |
| `LTV` | 用户生命周期价值 | LTV 均值计算 |

**前后链路分离规则**：
- **后链路指标**（线索数、首单数、首单流水、LTV）：月度汇总必须基于**完整的**本文件数据（不筛选 `TARGET_RESOURCES`），仅资源位级别分析才筛选目标资源位
- **前链路指标**（曝光UV、点击UV）：来自 `4月广告位明细.xlsx` / `APP广告位明细3月汇总.xlsx`

**注**：新文件已移除 `广告资源位`、`广告位素材`、`面向人群` 三列。`tag_level_1` 等效替代 `广告资源位`；`camp_name` 替代 `广告位素材` 用于卖点提取；`面向人群` 自动填充为 `'全部用户'`。

### 3.2 月活数据源

**文件**：`3-4月月活人数.xlsx`
**Sheet**：`用户等级月活人数`
**规模**：22 行（11 个用户等级 × 2 个月）

| 字段 | 说明 | 用途 |
|-----|------|------|
| `月份` | 2026-03 / 2026-04 | 月份切分 |
| `用户等级` | 0–10 | 等级拆分维度 |
| `月活人数` | 该等级当月活跃用户数 | MAU 计算与趋势图 |
| `占比` | 该等级占当月总 MAU 比例 | 辅助验证 |

### 3.3 前链路数据源

**文件**：`4月广告位明细.xlsx` / `APP广告位明细3月汇总.xlsx`
**用途**：提供曝光UV、点击UV、售卖页浏览UV 等前链路指标
**关键约束**：
- 广告位名称需通过 `AD_NAME_MAP` 映射为统一资源位名称
- 仅用于前端效率分析（CTR、CTCVR、线索生成率）
- 月度汇总的前链路指标（曝光UV、点击UV）来自本文件聚合

### 3.4 品类流量结构数据源

**文件**：`APP品类流量结构.csv`
**格式**：左右并排双表，左侧为 4 月数据，右侧为 3 月数据
**关键列**：`品类区别`（正式品 / 孵化品）

### 3.5 品类属性映射数据源

**文件**：`【重要】品类归属.xlsx`
**用途**：将 `category_name` 映射为品类属性（兴趣线/健康线/变美线）与品类情况（正式品/孵化品）
**关键约束**：精确匹配 + 包含匹配 fallback，未命中标记「未分类」

### 3.6 飞书多维表格数据源（周报专用）

**用途**：周报数据来源，提供比 Excel 更实时的前后链路数据
**约束**：
- Feishu Base 有 20,000 行限制，4 月前链路数据拆分为 `frontend_data_2026_04_p1` 和 `frontend_data_2026_04_p2`
- `lark-cli base +record-list --format json` 返回 2D 数组结构，字段名来自 `data.fields`
- `field-list` 与 `record-list` 字段可能不一致，读取时必须以 `record-list` 实际返回的字段为准，避免丢列
- `order_time` 字段用于周报按日筛选（MTD 范围：1 日至最新日）

## 4. 核心指标口径

### 4.1 产值拆解公式

```
资源位 GMV = 曝光UV × CTR × CTCVR × 首单转化率 × 客单价
```

复盘时更关注**归因视角**：
- 总 GMV 环比下降，是**哪个资源位**拖了后腿？
- 该资源位产值下降，是因为**线索量少了**，还是**单线索产出的钱少了**？
- 价格带结构是否恶化（0元课占比上升 → 整体 ARPU 被拉低）？

### 4.2 指标层级

| 层级 | 指标 | 用途 | 公式 |
|------|------|------|------|
| **产值结果** | GMV、线索数、首单数 | 结果呈现 | `SUM(首单流水)` / `SUM(线索数)` / `SUM(首单数)` |
| **后端效率** | 首单转化率（线索→首单）、单线索产出（LTV） | 产值归因核心 | `SUM(首单数) / SUM(线索数)` / `SUM(首单流水) / SUM(线索数)` |
| **前端效率** | CTR（点击/曝光）、CTCVR（领课转化率=线索/点击）、线索生成率（线索/曝光） | 流量入口效率 | `点击UV / 曝光UV` / `线索数 / 点击UV` / `线索数 / 曝光UV` |
| **结构健康** | 价格带占比（0元/1.1元/3.9元/其他）、品类类型占比（正式品/孵化品） | 结构诊断 | 该价格带线索数 / 当月总线索数 |

**完课率、到课率、好友率属于后链路过程指标**，可以在下钻时展示，但不应作为 headline 级别的核心诊断信号。

### 4.3 指标口径汇总表

| 指标名称 | 计算公式 | 说明 |
|---------|---------|------|
| **CTR** | `点击UV / 曝光UV` | 前端流量吸引效率 |
| **CTCVR（领课转化率/点击转化率）** | `线索数 / 点击UV` | 点击→线索的转化效率 |
| **线索生成率** | `线索数 / 曝光UV` | 综合入口效率 |
| **首单转化率（线索→首单）** | `SUM(首单数) / SUM(线索数)` | **唯一正确公式，严禁直接 AVG 原始 `转化率` 字段** |
| **完课→首单转化率** | `首单数 / 完课数` | 后链路过程指标 |
| **单线索产出（LTV）** | `GMV / 线索数` | 单个线索带来的产值 |
| **单曝光产出** | `GMV / 曝光UV` | 每次曝光的产值贡献 |
| **单点击产出** | `GMV / 点击UV` | 每次点击的产值贡献 |
| **环比** | `(本月值 - 上月值) / 上月值 × 100%` | 上月值为 0 时显示「—」 |
| **pp（百分点）** | `本月率 - 上月率` | 用于占比变化 |

## 5. 功能架构

### 5.1 看板结构：四屏 + 周视图

| 屏幕 | 标题 | 决策层级 | 目标 |
|-----|------|---------|------|
| 屏幕1 | 核心产值诊断 | 「发生了什么」 | 让管理层 30 秒看懂上个月 APP 产值变化 |
| 屏幕2 | 转化链路全景 | 「哪里坏了」 | 区分前端流量效率 vs 后端变现效率 |
| 屏幕3 | 根因下钻 | 「为什么坏了」 | 三维交叉定位产值下降的具体病灶 |
| 屏幕4 | 策略建议 | 「怎么办」 | 自动生成可落地的策略建议与行动清单 |
| **周视图** | **MTD 进度周报** | **「现在怎么样」** | **月度中期进度追踪、目标差距预警、同比归因** |

### 5.2 屏幕1：核心产值诊断

**目标**：让管理层 30 秒看懂「上个月 APP 产值怎么了」。

**布局**：
- **Headline**：基于真实数据的自动诊断
  - 示例：`4月 GMV 环比下降 10.9%（-¥23.7万），主因：选课中心产值下滑 ¥14.9万 + 学习页产值下滑 ¥10.3万`
  - 归因逻辑：GMV 变化最大的 Top 2 资源位作为 headline 主因
- **KPI 卡片（4 张，产值强相关）**：
  1. GMV（3月 → 4月，环比）
  2. 线索数（3月 → 4月，环比）
  3. 整体首单转化率（线索→首单，环比）
  4. 单线索产出（LTV，环比）
- **信号卡片（3 张）**：基于数据自动生成的关键变化信号
  - 例：`学习页单线索产出：¥84.8 → ¥56.4（↓33%），价格带结构恶化，0元课占比上升`
- **行动建议卡片（3 张）**：按负责方（产研/运营/内容）分类的 P0 行动项
- **次月行动清单**：P0/P1/P2 优先级行动卡片

**删除**：完课率、到课率 headline 信号卡片。

### 5.3 屏幕2：转化链路全景

**目标**：区分「前端流量入口效率」和「后端变现效率」，避免把 CTR 下滑和产值下滑混为一谈。

**布局**：左右分栏

**左侧：前端链路效率（曝光 → 线索）**
- 各资源位「CTR」对比柱状图（3月 vs 4月）
- 各资源位「CTCVR = 领课转化率 = 线索/点击」对比柱状图
- 各资源位「线索生成率」对比（线索/曝光）
- 作用：回答「是不是流量入口坏了」「哪个资源位的点击→线索效率在恶化」

**右侧：后端产值效率（线索 → GMV）**
- 各资源位「首单转化率」对比柱状图（3月 vs 4月）
- 各资源位「单线索产出（LTV）」对比柱状图
- 作用：回答「哪个资源位的线索不值钱了」

**下方：全宽诊断卡片（3 张）**
- 基于真实数据，自动找出变化最大的 3 个环节

### 5.4 屏幕3：根因下钻

**目标**：支持下钻到「资源位 × 价格带 × 品类类型」三维交叉，定位产值下降的具体病灶。

**布局**：

**模块1：资源位产值矩阵（全宽散点图）**
- 横轴：线索数（对数刻度）
- 纵轴：单线索产出（GMV/线索数）
- 气泡大小：GMV 规模
- 四象限：右上（明星）、左上（精准可扩量）、右下（粗放需调结构）、左下（观察淘汰）
- 点击交互：选中资源位，联动下方模块

**模块2：资源位 GMV 瀑布图（半宽）**
- 展示各资源位 3月→4月 的 GMV 变化量
- 直观看出「谁拖了后腿、谁补了缺口」

**模块3：选中资源位的价格带结构（半宽）**
- 默认展示整体，点击模块1 的某个点后，切换为该资源位的价格带饼图
- 3月 vs 4月 并排环形图
- 标注各价格带占比变化（pp）

**模块4：价格带全局结构（1/3 宽）**
- 3月 vs 4月 双环形图（真实数据）
- 点击效率矩阵联动：切换为选中资源位的价格带结构

**模块5：品类类型效率（1/3 宽）**
- 正式品 vs 孵化品 的线索数、转化率、GMV 对比
- 基于 `category_name` 与 CSV `品类` 的映射
- 点击效率矩阵联动：切换为选中资源位的品类类型结构

**模块6：用户等级效率（1/3 宽）**
- 横向堆叠柱状图（真实数据）
- 展示该资源位带来的线索分别来自哪些用户等级

**模块7：品类产出分析（全宽）**
- 品类汇总卡片（兴趣线/健康线/变美线）：线索数、GMV、转化率、单线索产出、正式品/孵化品对比
- 品类明细表：40 个品类，展示线索数、GMV、首单转化率、单线索产出、GMV/线索/CVR Top3 资源位
- 点击联动：点击品类行 → 下钻显示该品类的价格带结构饼图

### 5.5 屏幕4：策略建议

**目标**：基于屏幕3的下钻结果，自动生成可落地的策略建议，支持一键导出为汇报材料。

**用户明确：不需要「预期 ROI」模拟调配。**

**布局**：

**模块A：核心结论卡片（全宽，3 张）**
- 基于数据自动生成的 3 条核心结论（替代虚构的策略卡片）
- 结论生成逻辑示例：
  - 若某资源位线索量大但单线索产出低于均值 30% → 「XX 资源位线索质量偏低，建议排查价格带结构」
  - 若 0元课占比上升 > 5pp → 「低价引流结构加重，需关注高价值转化」
  - 若某资源位 GMV 环比下降贡献最大 → 「XX 资源位是产值下降主因，需专项复盘」

**模块B：资源位效率预警表（全宽）**
- 列：资源位 | 线索数环比 | 首单转化率环比 | 单线索产出环比 | GMV 贡献变化 | 健康状态
- 健康状态逻辑：
  - 若 CVR 环比下滑 ≥ 10% 或 单线索产出环比下滑 ≥ 15% → 「预警」（黄）
  - 若 GMV 贡献为正且 CVR 稳定 → 「健康」（绿）
  - 其余 → 「观察」（灰）
- 支持点击行下钻到该资源位的历史趋势

**模块C：次月核心行动清单（全宽）**
- 基于模块A 的结论，自动生成行动项（人工可编辑）
- 格式：序号 + 行动标题 + 负责方 + 截止日期 + 优先级
- 数据来源：`action_checklist` 由 Python 根据数据自动触发规则生成，而非硬编码

### 5.6 周视图：MTD 进度周报

**目标**：月度中期进度追踪，让运营快速知道「本月截至今天的进度 vs 目标差距」以及「同比哪里出了问题」。

**入口**：顶部导航栏「周视图」标签

**布局**：

**KPI 卡片（4 张）**
1. 线索数（MTD）：当前值、环比、目标、差距
2. GMV（MTD）：当前值（万）、环比、目标、差距
3. 转化率（CVR）：当前值、环比（pp）
4. 线索生成率：当前值、环比（pp）

**资源位线索变化 Top5（半宽表格）**

## 9. 技术实现

### 9.1 数据流

```
generate_analysis.py
  → 读取 Excel + CSV
  → 清洗、过滤合计行、计算指标
  → 输出 data_analysis_output.json

generate_dashboard_v2.py
  → 读取 data_analysis_output.json + weekly_report_data.json（如存在）
  → Python f-string 拼接 HTML（含内嵌 CSS + JS + RAW_DATA）
  → 输出 dashboard/v2/index.html
```

**每次数据或模板改动后，必须依次运行两个脚本。**

### 9.2 关键技术约束

- **单文件 HTML**：Tailwind CSS CDN + ECharts 5.5 + ECharts WordCloud 2.1.0
- **f-string 转义陷阱**：严禁在 f-string 内使用 `\n`、`\'`、`"` 等反斜杠转义序列；JS 对象字面量中的 `{` `}` 必须转义为 `{{` `}}`
- **转化率口径**：唯一正确公式 `SUM(首单数) / SUM(线索数)`，严禁直接 AVG 原始 `转化率` 字段
- **合计行过滤**：必须在任何计算前执行 `df = df[df['stat_month'] != '合计']`
- **Feishu Base 读取**：`record-list` 与 `field-list` 字段可能不一致，以 `record-list` 实际返回的 `data.fields` 为准构造 DataFrame，避免丢列
- **品类名称映射**：Excel `category_name` 与 CSV `品类` 优先精确匹配，未命中 fallback 到包含匹配，仍未命中标记「未分类」
- **ECharts 旭日图**：`label.formatter` 中严禁使用 `\n`（Python f-string 会将其转为真实换行，导致 JS 语法错误）

### 9.3 脚本依赖

```
generate_analysis.py
  ├── pandas, openpyxl
  └── 读取：Excel/CSV

generate_dashboard_v2.py
  ├── json, os
  └── 读取：data_analysis_output.json, weekly_report_data.json（可选）

generate_weekly_report.py
  ├── pandas, json
  ├── feishu_reader.py（可选）
  └── 读取：Excel 或 Feishu Base

generate_weekly_charts.py
  ├── build_chart（chart skill）
  └── 读取：weekly_report_data.json
```
